# 브랜치 전략 가이드

이 문서는 현재 저장소의 협업 브랜치 전략과 병합 순서를 정리합니다.

## 1) 현재 브랜치 구조

원격(`origin`):

- `main`
- `develop`
- `feature/ui`
- `feature/login`
- `feature/messenger`
- `feature/payment`

로컬(예시):

- `main`
- `develop`
- `feature/ui`

## 2) 브랜치 역할

- `main`: 운영 배포 기준 브랜치
- `develop`: 기능 통합/검증 브랜치
- `feature/*`: 기능 단위 작업 브랜치

## 3) 권장 작업 흐름

1. `develop` 최신화
2. `feature/*` 생성 후 개발
3. `feature/* -> develop` 병합(PR 권장)
4. 통합 검증
5. `develop -> main` 반영

## 4) 네이밍 규칙

- `feature/<주제>`: 기능 개발
- `fix/<주제>`: 버그 수정
- `hotfix/<주제>`: 운영 긴급 수정
- `chore/<주제>`: 설정/문서/빌드 작업

예시:

- `feature/ui`
- `fix/login-redirect`
- `chore/render-deploy`

## 5) 자주 쓰는 명령어

### 기능 브랜치 생성

```bash
git fetch origin
git switch develop
git pull origin develop
git switch -c feature/<name>
```

### 작업 반영

```bash
git add .
git commit -m "feat: <summary>"
git push -u origin feature/<name>
```

### feature/ui를 develop에 반영

```bash
git fetch origin
git switch develop
git pull origin develop
git merge origin/feature/ui
git push origin develop
```

## 6) 병합 정책

1. `main` 직접 커밋 금지(긴급 상황 제외)
2. `develop` 반영 전 로컬 빌드 확인
3. 배포 관련 파일 변경 시 별도 리뷰
4. 충돌 해결은 임시 브랜치에서 진행 권장

임시 브랜치 예시:

```bash
git switch -c merge/develop-to-main origin/main
git merge origin/develop
```

## 7) 병합 전 체크리스트

- `npm run build` 성공
- 백엔드 빌드 또는 Docker 빌드 확인
- 환경변수/API/CORS 변경사항 문서화
- 불필요한 산출물/임시 파일 제외 확인

## 8) 충돌 발생 시 원칙

1. 우선 `main` 안정성 유지
2. 충돌 파일은 담당자 기준으로 한 명이 최종 정리
3. 정리 후 변경 이유를 PR 설명에 명시
4. 머지 직후 재배포/핵심 기능 점검
